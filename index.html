Setup Guide

Getting Started

Everything you need to go from zero to your first Claude Code session.

route
Recommended order for this page
Read the mindset section first — it'll reframe how you approach the setup steps. Then follow the setup steps in order (Steps 1–4 before Steps 5–8). After your first session, head to Claude to understand why things worked the way they did.

The mindset before the tool

You've spent years learning to communicate design intent to other people — to engineers, stakeholders, junior designers. What you're about to learn is how to communicate that same intent to Claude Code.

The biggest mistake designers make with AI tools is treating them like search engines — typing a vague request and hoping for something useful. Claude Code works best when you bring the same specificity you'd bring to a design brief: here's the context, here's the constraint, here's what good looks like. The tool is powerful. Your judgment and direction are irreplaceable.

Key mental model: You're not learning to code. You're learning to direct code-generation with creative precision. That difference matters.

Set up your environment

Step 1 — Install Node.js

Go to nodejs.org and download the LTS version. Run the installer. Verify in Terminal:

node --version

You should see something like v20.11.0. Any version 18 or higher works.

Step 2 — Install Claude Code

npm install -g @anthropic-ai/claude-code

When it finishes, run claude --version to confirm.

Step 3 — Install VS Code

Download VS Code. Then press Cmd+Shift+P, type "Shell Command", and select "Install 'code' command in PATH". This lets you open any folder in VS Code from Terminal by typing code .

Step 4 — Authenticate Claude Code

claude

It walks you through connecting your Anthropic account. Takes about 2 minutes.

Step 4.5 — Show hidden files and pre-allow permissions

Claude stores its configuration in a .claude/ folder inside your project. By default, VS Code hides files that start with a dot. To see them: open Settings (Cmd+,), search "files.exclude", and remove the **/.* pattern if it exists. You'll need visibility into this folder later when you create skills and CLAUDE.md files.

Next, reduce permission interruptions. By default, Claude asks before every file read and bash command. For design work, pre-allow the safe ones:

Help me set up permissions. Pre-allow: reading any file without asking, and running ls, cd, cat, mkdir, and open without asking.

Claude will update your settings. You can always review and change these later in .claude/settings.json.

Your first session

Step 5 — Create a project folder

mkdir my-first-component
cd my-first-component
code .

Step 6 — Start a Claude Code session

claude

You'll see a > prompt. You're in a session. Claude has read your folder and is ready.

Step 7 — Give Claude your first directive

Create a single HTML file called index.html with a button component. Primary filled style in deep navy (#1a1f36), white label, 8px border-radius, clear hover state, pressed state that scales down slightly. Include a ghost variant. Show both on the page with labels. Pure HTML and CSS only — no frameworks.
stars
What good first output looks like
Claude should produce a working HTML file with both button variants visible on a white page, all states present (default, hover, active, focus, disabled), no external dependencies, and CSS custom properties for colors. If it added a JavaScript framework or used inline styles everywhere — those are signs to scope more tightly next time. Read the output file before iterating.

Step 8 — Iterate on the output

The hover state is too subtle — make it more pronounced. Add a destructive variant in red (#c0392b) matching the primary structure. Increase font size to 15px across all variants.
psychology
Good iteration is specific
The prompt above works because it names the problem (hover state is too subtle), describes the fix (more pronounced), and scopes additional asks clearly. Compare to "make it better" — which gives Claude nothing to work with. The more specifically you can describe a visual problem, the more precisely Claude can fix it.

When things go wrong

Claude made a change you didn't want: Type /undo in the session to revert. Or use git — commit often so you always have a restore point.

The output doesn't look right: Describe the visual symptom: "The button is stretching full-width — it should be inline, sized to its label." Claude understands design language.

Claude changed things you didn't ask about: Scope your sessions: "Only make changes to the button component. Do not modify anything else."

⚠️ Critical habit: Always read what Claude built before asking for the next thing. You'll catch misinterpretations early and your prompts will get sharper fast.

Foundations

Claude

What Claude is, how it thinks, and how to work with it effectively as a designer.

The Model

What Claude is

Claude is a large language model built by Anthropic. It reads text, understands context, and generates responses — including code, copy, analysis, and structured documents. It has no memory between sessions unless you give it context explicitly.

01
Language model
Trained on text. Understands and generates language — including code, which is just a formal kind of language.
02
No memory by default
Each session starts fresh. What you built last week doesn't exist unless you bring it in — through files, CLAUDE.md, or pasting context.
03
Context-sensitive
The more relevant context you give it — your project, your constraints, your visual language — the better the output matches your intent.
04
Not a search engine
It doesn't look things up in real time. It reasons from its training and your context. Treat it like a very capable collaborator, not Google.
error_outline
Common misconception
Claude doesn't improve between sessions. Giving it better context in this session is what gets better output — not hoping it "learned" from the last one. Every session is a fresh start. Your CLAUDE.md is how you transfer what it knows.
The Tool

What Claude Code looks like

Claude Code runs in your terminal — the same terminal you use for git, npm, or any other developer tool. You type claude inside a project folder and start a conversation. Claude reads your files, proposes changes, and writes code directly into your project.

What it can do

01
Read your project
Claude sees your entire file structure — every component, stylesheet, token file, and config. It understands how your project is organized and what already exists.
02
Write and edit files
Claude doesn't just suggest code — it writes directly into your files. You see a diff of every change before accepting it. Nothing is permanent until you approve.
03
Run commands
Claude can run terminal commands — start a dev server, install packages, run tests, commit to git. It asks permission before running anything destructive.
04
See images
Drag a screenshot into the terminal and Claude can see it. This is how design review works — paste a screenshot, ask Claude to evaluate it against your design principles.

What to expect

terminal
It's text-based, not visual
There's no GUI, no drag-and-drop, no canvas. You communicate through typed prompts and Claude responds with text and file changes. This feels unfamiliar at first — but it's faster than you'd expect. Most designers adapt within a few sessions.
edit_note
You direct, Claude executes
Think of it like briefing a very fast junior designer who happens to write code. You describe what you want — layout, spacing, color, behavior — and Claude builds it. Your design vocabulary is the interface.
visibility
You always see what changed
Every change Claude proposes shows as a diff — green lines added, red lines removed. You review, accept, or reject each one. You're never surprised by what happened to your files.
undo
Everything is reversible
Type /undo to reverse the last change. Combine with git commits and you have unlimited restore points. There's no way to permanently break something if you're committing regularly.
lightbulb
You don't need to know how to code
Claude Code writes the code. Your job is to describe what you want with precision — the same skill you use when briefing an engineer or writing a design spec. If you can describe spacing, color, layout, and interaction in words, you can use Claude Code.
The Loop

How a session works

A Claude Code session is a conversation that happens inside your project folder. Claude reads your files at the start of each session. You prompt, it proposes changes, you accept or reject. That's the loop — everything else is variation on it.

terminal
Open your project in terminal
cd my-project then claude. Claude reads your file structure and any CLAUDE.md files it finds. It now has context for everything in that folder.
edit_note
Write a prompt
Describe what you want built, changed, or reviewed. The more context you include — visual intent, constraints, what already exists — the better the output.
difference
Review the diff
Claude shows exactly what it changed — every line added, removed, or modified. Read this before accepting. This is where you catch misinterpretations early.
check_circle
Accept or reject
Accept writes the changes to your files. Reject discards them. You can also use /undo at any point. Nothing is permanent until you accept it.
refresh
Iterate
Prompt again. Claude remembers everything from earlier in the session. Keep going until the component is right — then commit to git as a restore point.
lightbulb
The diff is your creative control panel
Every change Claude proposes shows as added lines (green) and removed lines (red). Read it before accepting. Designers who skip the diff lose creative control — designers who read it catch misinterpretations before they compound. It takes 30 seconds and it's the most important habit in the workflow.

Oversight

Plan mode and oversight levels

Claude Code has three oversight modes. Press Shift+Tab at the prompt to cycle between them. The right mode depends on how much you trust what's about to happen — match oversight to risk, like supervising a junior designer.

Plan
Explore before acting
Claude proposes an approach without writing any code. Use this for big decisions, unfamiliar territory, or when you want to align on direction before execution. Read the plan critically — a plan you don't review gives false confidence.
Default
Review every change
Claude proposes changes and waits for you to accept or reject each one. This is your standard working mode — you see the diff, decide whether it's right, then move on. Most sessions should spend most of their time here.
Auto
Trust and execute
Claude applies changes without asking. Use this only for trusted mechanical tasks — renaming variables, formatting, or applying a pattern you've already validated. Never use it for design decisions.
Key habit: Flip between modes constantly during a session — not once at the start. Start in plan mode to align on the approach, drop into default for the build, switch to auto for the cleanup. The overhead of switching is seconds; the cost of the wrong mode is wasted work.
Session Management

Parallel sessions, naming, and resuming

Each Claude Code session is an isolated context. Mixing unrelated tasks in one session degrades output for all of them. The solution is multiple focused sessions — and tools to manage them.

splitscreen
Parallel sessions
Open separate terminal tabs in VS Code, each running its own claude instance. One session builds the component, another writes the spec, a third explores a layout approach. Each gets Claude's full attention — no cross-contamination.
label
Naming sessions
Type /rename Dashboard Cards to tag the current session. Named sessions are easy to find later. Without a name, sessions blur together and you'll lose track of what's where.
history
Resuming sessions
Type /resume to browse past sessions, or /resume Dashboard Cards to jump directly. From outside Claude, run claude --resume "Dashboard Cards" to start right where you left off. Context is preserved — Claude still remembers the decisions from last time.
task
Background agents
Delegate side work without leaving your main session. Tell Claude: "Compile all the color tokens into a reference table — do this in the background." Type /tasks to check on progress. The background agent gets a copy of your context and works independently.
lightbulb
Split your terminal, not your attention
In VS Code, split the terminal panel to see 2-3 sessions side by side. Right-click each terminal tab and rename it to match your Claude session name. This way you always know which session is which without clicking through tabs.
Prompting

How to write a good prompt

Most bad output comes from underprompted input. A good prompt has four parts — and most people include one. The more of these you give Claude, the less time you'll spend correcting the wrong thing.

task_alt
The task
What you want Claude to do. Be specific: not "build a card" but "build a data summary card with a metric value, label, and percentage change indicator."
info
The context
What already exists. Which files are relevant. What system you're working in. What the component sits next to. Claude can't assume — it only knows what you tell it.
block
The constraints
What rules apply. Tokens only — no hardcoded values. Match the existing component library. No shadows. These are your design guardrails — include them or Claude will fill the gap with its own defaults.
output
The output format
What you want back. A single file? A component with all variants? A list of decisions it made? When you don't specify, Claude decides — and often gives you far more than you asked for.
Weak prompt
Build me a button component
No context. No constraints. No output format. Claude will guess at the style, the states, the file structure, and whether to match anything that already exists.
Strong prompt
Build a primary button for the dashboard
Use tokens from tokens/colors.css. Four states: default, hover, focus, disabled. No shadows — use border only for state differentiation. Single file output. Match the visual density of the existing nav component.
lightbulb
Describe your design intent, not just the output
"Make the header larger" tells Claude what to change. "The header needs more visual weight — the current size reads as body copy, not a section title" tells it why, and it'll choose the right combination of size, weight, and spacing to solve it. Intent-first prompts produce better output than instruction-only prompts.

Iteration

How to iterate and correct

Prompting is starting. Iteration is how you close the gap between Claude's first attempt and your actual intent. The right correction vocabulary makes the difference between a session that converges and one that spirals.

settings_backup_restore
Correction — "Keep X, change Y"
When some of the output is right, name what to preserve first. "Keep the layout and colors. Change only the spacing between label and value to 4px." Without this, Claude may re-interpret everything.
tune
Refinement — describe what's wrong, not what to do
"The button feels visually heavy" often produces better output than "make the button smaller." Tell Claude the design problem — it frequently knows the right combination of fixes better than a single specific instruction.
compare_arrows
Comparison — "like X but Y"
"Like the existing nav component but with smaller type and 8px vertical padding." Reference something in your project that already works. It's faster and more precise than describing the thing from scratch.
add_circle
Adding constraints mid-session
"Going forward in this session: never use box-shadow — use 1px border instead." Claude applies this to everything after. This is how you refine your CLAUDE.md live — discover what bothers you, then lock it in as a rule.
not_interested
Pattern override — stop doing X entirely
When Claude repeats the same mistake: "Stop. You keep adding shadows. This project never uses shadows — only 1px borders for elevation. Go back and fix every instance." Be direct and comprehensive.
call_split
Delegate side work — "do this in the background"
When a side task would break your flow: "Generate a spacing reference table from our tokens file — do this in the background." Claude spawns a background agent that works independently while you stay focused. Check on it with /tasks.
lightbulb
Build your system as you go
After every session, ask yourself: "Will I need to explain this again?" If yes, add it to your CLAUDE.md. If you wrote a prompt that worked well, save it. If you corrected the same mistake three times, add it to anti-patterns. The compounding value of Claude Code isn't in any single session — it's in the system you build around it. Each good prompt you save is one you never have to write again.
error_outline
Don't ask why — tell Claude what to do differently
"Why did you use that padding?" puts Claude in explanation mode. When output is wrong, skip the analysis and direct the fix: "The padding is wrong — 16px horizontal, 10px vertical." This is faster, more useful, and keeps the session in productive territory. You can ask why after the session if you're curious — during it, it just costs context.
checklist
Quick-reference: iteration phrases that work
To preserve good output: "Keep everything except…"
To correct direction: "That's not the right approach — instead…"
To escalate a recurring issue: "This is now a rule: never [X]."
To reference existing work: "Match the [component name] pattern exactly."
To get variants: "Give me three approaches — light, medium, and heavy visual weight."
To zoom out: "Pause. Before continuing, tell me the three design decisions you're about to make."
Memory & Limits

Context windows

A context window is the total amount of text Claude can hold in "working memory" at once — your prompt, the conversation history, and every file it's read. When a session gets long, earlier parts fall out of context. Claude isn't forgetting — it simply can't see that far back. The window isn't just storage — it's also where Claude reasons. A full window means worse thinking, even if the information is still technically there.

Monitor your usage

Type /statusline in any session and tell Claude what to display — your model name, working directory, and context usage as a percentage or token count. This gives you a live readout at the bottom of your terminal so you can see how full the window is before it becomes a problem.

0–50%
Spacious
Work freely on any task. Claude has plenty of room to reason through complex problems. This is where you want to be for creative work and design decisions.
50–70%
Narrowing
Stick to straightforward tasks. Claude can still work but has less room for nuance. If you're about to ask something complex, consider compacting first.
70%+
Auto-compaction zone
Claude will soon auto-summarize earlier conversation to make room. When it does, detailed instructions become bullet points and nuance gets lost. Don't let this happen to you — compact manually before Claude does it for you.

Manual compaction

Instead of letting Claude auto-compress your context (and decide what to keep), do it yourself. Ask Claude: "What do you currently have in context from our conversation?" Review what's there, then run:

/compact Keep the current component structure, design tokens, and the spacing decisions we made. Drop the early exploration and the first two approaches we rejected.

After compacting, press Ctrl+O to view the summary Claude kept. Verify that your critical context survived. This takes 30 seconds and prevents the most common quality problem in long sessions.

When to start fresh vs. compact

Start fresh
You're switching to a completely different task. Output feels generic despite specific prompts. Claude repeats mistakes you already corrected. You've been in the same session for 2+ hours.
Compact
You're staying on the same task but the window is filling up. You want to keep key decisions and constraints but drop the exploration that led to them. You're at 50-60% and about to do something complex.
Use /clear
You want to switch tasks within the same session. /clear wipes the conversation but keeps your CLAUDE.md loaded. Useful when you want a fresh start without re-authenticating or reloading your project context.
Commit as you go. Commit to git after every meaningful change. Each commit is a restore point. If a session goes sideways — or you compact and lose something important — you can always roll back to the last good state.
The Brief

What CLAUDE.md is

CLAUDE.md is a markdown file at the root of your project that Claude reads at the start of every session. It's your standing creative brief — visual language, conventions, constraints, and anti-patterns. Without it, Claude works from generic defaults. With it, Claude works from your system.

info
Project overview
What this is, who it's for, what success looks like.
palette
Visual language
Color approach, type personality, spacing density, border radius, motion feel.
widgets
Component conventions
How buttons, cards, forms, and navigation should behave in this project.
block
Anti-patterns
What not to do. Specific patterns Claude should never generate for this project.
CLAUDE.md 
# Project Overview
Dashboard for data operations teams.
Dense UI, high information density.
Audience: power users, daily use.

# Visual Language
Color: Navy primary (#013D5B)
Type: 14px base, tight line-height
Spacing: Base 4px, compact
Radius: 4px small, 6px components

# Anti-Patterns
- No rounded pill buttons
- No card shadows (use borders)
- No font below 12px
- No hardcoded hex values
Think of CLAUDE.md as the design brief you never have to repeat. Once it's written, every session starts with Claude already knowing your system.

The hierarchy: global → project → folder

CLAUDE.md files stack. When you start a session, Claude loads every CLAUDE.md it finds from your home directory down to the current folder. Each level adds more specific context:

person
~/.claude/CLAUDE.md — Personal preferences
How you like to work. Communication style, formatting preferences, general patterns. Applies to every project. Keep it short — one screen maximum.
folder
~/my-project/CLAUDE.md — Project context
Design system, visual language, tech stack, team conventions. This is your main creative brief — the one that defines how this project looks and behaves.
subdirectory_arrow_right
~/my-project/components/CLAUDE.md — Folder-specific
Rules for this area only. Component naming conventions, required states, accessibility patterns. Only loads when you're working inside this folder.

You can reference other files from within CLAUDE.md using @./path/filename.md. This lets you include a brand voice doc, a token reference, or a component checklist without duplicating content.

Three ways to create one

Interview method
Best when context is in your head
Switch to plan mode and tell Claude: "Interview me about this project, then draft the CLAUDE.md based on my answers." Claude asks about scope, audience, visual language, and constraints. Answer conversationally — Claude writes the structured file from your responses.
From existing docs
Best when you already have specs
Drop your design briefs, wikis, or style guides into the project folder. Then: "Read the files in /docs and generate a CLAUDE.md. Focus on what you'd need to work effectively here." Review the output and trim — Claude tends to include background that doesn't help it work.
priority_high
Keep it lean — every line costs thinking room
CLAUDE.md loads into the context window at the start of every session. A 3-page file wastes thinking capacity before you've even typed your first prompt. Include only what applies to 80%+ of sessions. Finished project details, historical decisions, and reference material belong in separate files — not in the always-loaded brief. Start with anti-patterns: it's easier to articulate what you hate than what you love, and Claude follows prohibitions more reliably than vague style preferences.

Reusable Prompts

What a skill is

A skill is a saved prompt template stored as a markdown file in .claude/commands/. Type /skill-name in any session and it runs — with your project context already loaded, and your variable input passed via $ARGUMENTS.

info
Build skills after you've built prompts
Don't write a skill on your first week. First, prompt ad-hoc until you find yourself writing the same structure over and over — then that's your skill. The best skills are distilled from prompts that already worked, not written from scratch. A skill you haven't needed three times yet is probably too speculative to maintain.
badge
Description
One line that appears in the /skills menu. This is how team members know what the skill does before running it.
input
$ARGUMENTS
The variable part — what changes each time you run the skill. Everything else in the file is fixed instruction.
rule
Fixed instructions
The standing logic — requirements, constraints, output format — that applies to every run of this skill.
output

Environment

The three-pane workspace

Multiple guides and experienced Claude Code designers converge on the same physical setup. Three panes, side by side, always visible. This isn't a suggestion — it's the single most impactful environment change you can make.

terminal Terminal — Claude Code
Where you prompt, direct, and iterate
This is your primary interface. You type prompts here, review diffs, approve or reject changes, and steer the session.
code Editor — VS Code / Cursor
Where you see the files change
Watch the code appear and change in real time as Claude works. You don't need to write code here — but seeing it builds understanding over time.
web Browser — Live preview
Where you see the visual result
A live-reloading browser showing what Claude just built. This is your design eye — you evaluate the output visually, then go back to the terminal to refine.
lightbulb
The loop is physical
Your eyes move left-to-right in a constant cycle: prompt (terminal) → watch (editor) → evaluate (browser) → refine (back to terminal). This physical rhythm is the session loop made visible. If you can't see all three at once, you'll break the loop.
On a single monitor, use a tiling window manager or macOS Split View. On a laptop, dedicate the full screen to these three panes and close everything else. A second monitor is ideal — terminal + editor on the main display, browser on the secondary.
First Workflow

Design review with Claude

Every guide we've reviewed agrees on this: design review is the highest-value, lowest-risk way to start using Claude Code. You don't need to generate anything. You just ask Claude to look at what exists and tell you what's off.

The screenshot review loop

This is the core workflow. It uses Claude's vision capabilities to compare visual output against your design principles.

photo_camera
1. Capture
Take a screenshot of the UI you want reviewed — a component, a full page, or a specific state. Drag the image into the terminal or paste the file path.
rate_review
2. Prompt with criteria
Don't just say "review this." Give Claude your lens: "Review this against our spacing system. Are we using 8px increments consistently? Flag anything that looks off."
checklist
3. Review findings
Claude returns a structured list of issues — spacing inconsistencies, color contrast problems, alignment errors, missing states. Evaluate each one with your design judgment.
build
4. Fix and re-check
Ask Claude to fix the issues it found, then screenshot again to verify. This build → screenshot → verify → iterate loop is the design equivalent of test-driven development.

What to review

01
Spacing consistency
Are margins and padding using your token scale? Is the vertical rhythm consistent? Are there any hardcoded pixel values that should be tokens?
02
Color & contrast
Do all text-on-background combinations meet WCAG AA (4.5:1 body text, 3:1 large text)? Are interactive elements distinct from static ones?
03
Component states
Does the component have all its states? Default, hover, focus, active, disabled, loading, error, empty. Missing states are the #1 gap in AI-generated UI.
04
Responsive behavior
How does it look at mobile, tablet, desktop? Are touch targets at least 44×44px? Does the layout collapse gracefully or break?
lightbulb
Make it a skill
Once you've done a few reviews manually, save the prompt as a skill: /design-review. Include your design principles, your token system, and your quality bar. This is the single most recommended first skill across every guide we reviewed.
Confidence Ladder

When to use what

Not every design task belongs in Claude Code. The best designers use it selectively — starting with low-risk activities and expanding as confidence grows. Here's the progression that works.

Start here
Design critique & review
Paste screenshots, get structured feedback against your design principles. Zero risk — Claude doesn't change anything. You're just getting a second opinion.
Week 1–2
Build confidence
Understanding the codebase
Ask questions: "What warning states do we currently have?" "How is the modal component structured?" Make better design decisions grounded in what actually exists.
Week 2–3
Start building
Prototyping
Design the happy path in Figma, then use Claude to build a working prototype. Start with isolated components, then graduate to full page layouts.
Week 3–5
Expand scope
0-to-1 exploration
Skip static mockups entirely. Generate multiple design directions as working code. Evaluate them in the browser, not on a Figma canvas. This is where the workflow fundamentally changes.
Week 5–8
Full fluency
Component generation & system governance
Generate production-ready components using your design system. Audit token usage. Enforce consistency across the codebase. Hand engineers draft PRs instead of just mockups.
Week 8–12
error_outline
When Claude Code is the wrong tool
Don't force everything through Claude. Stay in Figma for: complex illustration and visual design, user flow mapping and journey diagramming, stakeholder presentation decks, and anything where you need to sketch loosely before committing to structure. Claude Code excels at structured, systematic work — not freeform visual thinking. That said, Claude is surprisingly useful in early ideation for research, teasing out concepts, and pressure-testing assumptions before you start designing.

Avoid These

Common pitfalls

These mistakes come up in almost every guide and tutorial we reviewed. They're not theoretical — they're the patterns that trip up real designers in their first weeks with Claude Code.

Writing one massive prompt
→
Work iteratively — one piece at a time
Designers often write a full-page brief expecting Claude to build an entire application. This produces generic, unfocused output. Instead: describe one component, review it, refine it, move to the next. Small prompts, fast loops.
Being too vague
→
Be specific about what you want
"Make it look better" gives Claude nothing to work with. "Increase the padding between cards to 24px, reduce the border-radius to 8px, and use our secondary color for the icon" gets you exactly what you asked for. Specificity is the skill.
No CLAUDE.md
→
Set up your project context first
Without a CLAUDE.md, Claude generates generic components with default styling. With one, every component uses your tokens, follows your conventions, and matches your system. This is the single biggest quality lever.
Skipping plan mode
→
Always plan before you build
Letting Claude jump straight to implementation is like letting a contractor start building before approving the blueprints. Hit Shift+Tab to enter plan mode, review the approach, then switch to implementation. This catches wrong directions before they cost you time.
No verification loop
→
Always check the visual output
Code that looks correct in a diff can look wrong in the browser. Always check the live preview after every change. Better yet, ask Claude to screenshot and self-check its own output before you even look at it.
Kitchen-sink sessions
→
One task per session, or /clear between tasks
Jumping between unrelated tasks — building a nav component, then debugging a form, then writing copy — fills the context window with noise. Each topic dilutes the next. Use /clear to reset between distinct tasks, or open parallel sessions.
Saying what you don't want
→
Say what you do want
"Don't use rounded corners, don't make it too colorful, don't use that font" gives Claude constraints but no direction. "Use 4px border-radius, a muted palette anchored on neutral grays, and Proxima Nova for all type" gives Claude a clear target. Positive instructions outperform negative ones every time.
Accepting without reviewing
→
Read the diff, check the browser
Auto-accepting feels fast but costs you control. The diff is where you exercise creative judgment. If you stop reading diffs, you stop directing — and the output drifts from your vision. Slow is smooth, and smooth is fast.

Most pitfalls share a root cause: treating Claude like a magic button instead of a collaborator. The designers who get the best results invest time in setup (CLAUDE.md, skills, workspace), work iteratively (small prompts, fast reviews), and maintain creative control (reading diffs, checking previews). 01
Most commands are read-only
The commands you'll use most — cd, ls, pwd — just look at things. They can't break anything. They're like opening Finder windows.
02
Claude asks before acting

03
Everything is reversible
Combined with git (which you'll learn next), you can always undo any change. There's no permanent damage possible when you're committing regularly.
04
You already use text interfaces

The Basics

What is the terminal?

The terminal is a text-based interface for talking to your computer. Instead of clicking folders and buttons, you type short commands.

On Mac, the app is called Terminal — you'll find it in Applications → Utilities, or search for it with Spotlight.

When you open Terminal, you'll see a prompt — usually ending in $ or %. That's where you type. Press Enter to run a command.

Think of your computer as a building

Your file system is like a building with rooms. The root directory (/) is the main entrance. Each folder is a room, and each / in a path is walking through a door into the next room.

home
Absolute path — full address from the entrance
/Users/yourname/Desktop/my-project is the complete address — it works no matter where you currently are, like a full street address.
near_me
Relative path — directions from where you stand
./src/components means "from the room I'm in, go through the src door, then the components door." The dot (.) means "here" and .. means "go back one room."
lightbulb
The #1 tip for visual thinkers
Drag a folder from Finder directly into the Terminal window — it pastes the full path for you. Type cd (with a space), then drag the folder. No typing long paths, no typos.

Print working directory. Tells you where you are right now. If you're ever lost, this shows the full path to your current folder.

keyboard_tab

Understanding

Reading what the terminal tells you

Error messages are not failure — they're diagnostic information. Learning to read them is the single biggest unlock after learning the basic commands.

The "read the last line first" habit

When the terminal shows a wall of text after an error, don't read from the top. Scroll to the last line — that's usually the actual problem. Everything above it is context about where the problem occurred.

Common messages you'll see

search_off
command not found
The tool isn't installed on your machine yet. Not an error with your project — you just need to install whatever it's asking for. Claude Code can help: "How do I install Node.js?"
lock
permission denied
You don't have access to that file or folder. Do not add sudo unless you understand exactly why. Claude Code should never require sudo for normal operation.
lan
EADDRINUSE or port already in use
Another process is already using that port. You probably have a dev server running in another terminal tab. Find it and stop it with Ctrl+C, or use a different port.
folder_off
no such file or directory
You're pointing to something that doesn't exist. Usually a typo in the path or you're in the wrong folder. Run pwd to check where you are and ls to see what's available.
bookmark
How to read the terminal prompt
When you see $ or % at the end of a line, the terminal is ready for your next command. If there's no prompt and text is still appearing, a command is still running. If you see a blinking cursor but no prompt, the terminal might be waiting for input — try pressing q or Ctrl+C.

Localhost is your computer talking to itself. When you see http://localhost:3000, your project is being served locally — no internet required. The number after the colon is the port, like a room number in a hotel. Your computer is the hotel, and each running server occupies a different room.

Concept

Package managers

npm and yarn are like app stores for code libraries. When a project needs external tools or frameworks, a package manager downloads and installs them.

01
package.json
The shopping list. This file lives in the project root and lists every library the project depends on. You don't edit this directly — tools manage it for you.
02
node_modules/
The downloaded apps. This folder can be huge (thousands of files) and is regenerated by running npm install. It's never committed to git.
03
npm install
Download everything on the shopping list. Always run this first when opening a project for the first time or after pulling updates. It reads package.json and fills node_modules/.
04
npm run [script]
Run a named recipe. The scripts section in package.json is a recipe book. npm run dev means "follow the recipe called dev." You don't need to memorize the full command.
bookmark
The one habit to remember
When you open a new project: npm install first, then npm run dev. Always in that order. Claude Code handles most package management for you, but knowing this sequence prevents the #1 beginner mistake — trying to run a project before installing its dependencies.

Git

Version control basics that keep your work safe when using Claude Code.

Why It Matters

Your safety net

Git is a version control system — think of it as unlimited undo for your entire project. It's also the single most important tool for working confidently with Claude Code.

Every time Claude makes changes to your files, those changes are real — they're written to disk. Git is your safety net. If Claude goes in a wrong direction, you can undo everything in seconds. If you like what Claude built, you can save a permanent checkpoint.

30s
Cost of a commit
A git add . and git commit -m "message" takes about 30 seconds. That's the price of a permanent save point you can always return to.
30m+
Cost of not committing
Without git, recovering from an AI-generated change that went wrong means manually undoing work file by file. That can take 30 minutes to hours — or be impossible.
bookmark
The confidence multiplier
When you know you can undo anything, you're willing to try bolder prompts, larger changes, and more experimental approaches. Git transforms "what if this breaks everything?" into "let's try it and see." That psychological shift is the real value.
Translation

Git in Figma terms

You already understand version control — you use it in Figma every day. Git is the same concepts with different names.

history
Commits = named versions in Figma's version history
When you create a named version in Figma ("Final header layout"), that's exactly what a git commit is. A labeled snapshot you can go back to anytime.
account_tree
Branches = Figma branches
Just like creating a Figma branch to explore a design direction without affecting the main file, git branches let you experiment in a parallel copy of your project.
rate_review
Pull requests = requesting a design review
When you share a Figma branch and ask for feedback before merging, that's a pull request. It's a review checkpoint, not a technical operation.
folder_special
Repository = your Figma file (with perfect memory)
A repo looks like a normal folder, but it remembers every version of every file forever. Unlike Figma, the history never expires or gets pruned.
compare_arrows
Merge conflict = two people edited the same layer
Git can handle two people editing different files automatically. But if two people change the same line, a human has to decide which version to keep — just like resolving conflicting changes in Figma.
Mental Model

How git thinks

Git tracks your project as a series of snapshots — like a time machine that can visit any point in your project's history. Understanding these four concepts makes everything else click.

01
Working directory
Your project folder — the files as they exist right now. When Claude edits files, this is what changes. Think of it as your drafting table.
02
Staging area
A packing box for your next checkpoint. You choose which changes go in the box before shipping it. Like selecting which artboards to include in an export — you don't have to include everything.
03
Commit
A saved checkpoint — a photograph of your entire project at one moment in time. Each commit has a message describing what changed. You can always travel back to any photograph.
04
Repository
Your project folder once git is tracking it. A repository (repo) is just a folder with a hidden .git directory that stores all your commit history.

Git vs GitHub

Git is a tool that runs locally on your computer. It tracks changes and stores history — no internet required. GitHub is a cloud platform that hosts your git repositories online so you can share them with your team. Think of it like the difference between saving a file on your desktop (git) and uploading it to Dropbox (GitHub).

Six commands that cover 95% of what you'll need. You can always look up the rest later.

Start tracking a project. Run this once in a new project folder to turn it into a git repository. After this, git will watch for changes.

See what changed. Shows which files have been modified, added, or deleted since your last commit. Use this before and after Claude sessions to understand what happened.

Stage all changes. Moves all your modified files into the staging area, ready to be committed. The dot means "everything in this folder."

Core Practice

The checkpoint habit

One simple rhythm will keep your work safe through every Claude Code session. This is the most important thing on this page.

The two-commit workflow

looks_one
Before starting Claude — commit your current state
git add . && git commit -m "checkpoint: before Claude session" — this is your restore point. If anything goes wrong, you can always come back here.
looks_two
Work with Claude — prompt, review, iterate
Claude makes changes to your files. You review diffs, accept or reject. When you're happy with the result, move to the next step.
looks_3
Review and commit the result
Run git diff to see everything Claude changed. Then git add . && git commit -m "Add responsive navigation". Now both the before and after states are saved permanently.

Checkpoints vs commits

Claude Code automatically captures file state before each edit, letting you undo within a session. But these checkpoints are ephemeral — they disappear when the session ends. Git commits are permanent. Think of checkpoints as local undo (Cmd+Z) and commits as named saves (File → Save As). Both are valuable — use them together.

bookmark
git diff is your most important AI skill
After Claude makes changes, git diff shows you exactly what was added (green/+) and removed (red/-). This is how you maintain understanding of your own project even when AI is writing the code. Reading diffs is a visual skill — designers pick it up quickly.
Advanced

Experimenting safely

Branches let you create a parallel universe where you can experiment freely. If the experiment works, merge it back. If not, discard it. Zero risk to your working project.

When to branch

Create a branch before asking Claude Code to attempt something ambitious — a large refactor, a new feature, or a risky experiment. If the result isn't what you want, you can simply delete the branch and your main project is untouched.

add_road
git checkout -b experiment-name
Create a new branch and switch to it. Everything you do from here is in the parallel universe. Your main branch is untouched. Name it something descriptive: git checkout -b try-new-nav.
swap_horiz
git checkout main
Switch back to your main branch. Your files instantly return to their main-branch state. The experiment still exists on its branch — you can switch back anytime.
merge
git merge experiment-name
Bring the experiment into main. If you liked the result, switch to main and merge. All the changes from your experiment branch are now part of your main project.
lightbulb
The experiment didn't work?
Just switch back to main: git checkout main. Your main branch is exactly as you left it. You can delete the experiment branch with git branch -d experiment-name or just leave it — unused branches don't take up meaningful space.
Recovery

When things go wrong

Git almost never destroys data. Here's how to recover from the most common mistakes — bookmark this section.

undo
"I want to undo everything Claude just did"
git checkout . — reverts all files back to your last commit. This is the nuclear undo. Only works for uncommitted changes — which is why the checkpoint habit matters.
replay
"I want to undo a specific commit"
git revert <hash> — creates a new commit that undoes a previous one. Find the hash with git log --oneline. This is safe because it doesn't erase history.
edit_off
"I committed but forgot a file"
git add forgotten-file then git commit --amend — adds the file to your last commit instead of creating a new one.
wrong_location
"I committed to the wrong branch"
git reset HEAD~1 — moves the commit back to your working directory (changes are preserved, not lost). Then switch to the correct branch and recommit.
key_off
"I accidentally committed secrets or .env"
git reset HEAD~1 to undo the commit, add the file to .gitignore, then recommit without the sensitive file. If already pushed to GitHub, consider the secret compromised and rotate it.
help
"I'm in a weird state and nothing works"
Run git status first — it always tells you where you are and suggests what to do next. If you're truly stuck, git stash saves your current changes aside, giving you a clean slate. You can bring them back later with git stash pop.

Context Engineering

The quality of Claude's output is shaped more by what it knows than what you ask. Learn to engineer the full context — not just the prompt.

The Shift

Beyond prompting

Most designers focus on writing better prompts. But prompting is just one input into Claude's output. The real lever is context engineering — shaping everything Claude knows when it works for you.

A mediocre prompt with great context (well-written CLAUDE.md, design tokens, skills) consistently outperforms a brilliant prompt with no context. The designers getting the best results aren't prompt wizards — they've built systems that give Claude the right knowledge at the right time.

01
Context > prompting
A prompt lives for one turn. Context (CLAUDE.md, skills, files) compounds across every session. Investing one hour in your CLAUDE.md saves dozens of hours of repeated prompting.
02
Think systems, not messages
You already know this from design systems — build the system once, get consistency forever. Context engineering is design systems thinking applied to AI collaboration.
03
Four layers, ranked by impact
CLAUDE.md (always-on) → Skills (on-demand) → Conversation management (session-level) → Prompts (per-turn). Most designers over-invest in the last layer and under-invest in the first three.
04
Encode taste, not just instructions
The best context engineers don't just tell Claude what to do — they encode their design judgment. Anti-patterns, visual language, quality bars. Claude produces output that feels like theirs.
Architecture

The context stack

Every piece of information Claude uses comes from one of these four layers. Understanding them — and investing in the right ones — is the core skill of context engineering.

foundation
Layer 1: CLAUDE.md — your always-on design brief
Loads automatically every session. Contains your project overview, visual language, design tokens, component conventions, and anti-patterns. This is the single highest-impact investment — one file that makes every session better. Keep it under ~100 lines. For each line, ask: "Would Claude make a mistake without this?"
build
Layer 2: Skills — on-demand workflows and knowledge
Markdown files in .claude/skills/ that load only when relevant. Put detailed procedures, reference material, and multi-step workflows here — not in CLAUDE.md. No context cost when not in use. A /design-review skill with your exact quality criteria is worth more than a hundred review prompts.
forum
Layer 3: Conversation — session-level management
How you manage the session itself: when to compact, when to /clear, when to start fresh, how to maintain focus. A focused session with good compaction produces dramatically better output than a sprawling one. Think of it as managing Claude's working memory.
chat
Layer 4: Prompts — per-turn instructions
The individual messages you type. Important, but the least leveraged layer — a prompt only lives for one turn. When your first three layers are strong, your prompts get shorter and simpler because Claude already knows your system, your patterns, and your quality bar.
bookmark
Where to invest your time
Most designers spend 90% of their effort on Layer 4 (prompts) and 10% on everything else. Flip that ratio. Spend your first week on CLAUDE.md (Layer 1). Your second week on your first skill (Layer 2). Learn compaction habits (Layer 3). By week three, your prompts almost write themselves — because Claude already knows what you need.
Framework

Structuring your prompts

Not every prompt needs structure, but when results aren't landing, this framework helps. Think of it like wireframing your prompt before you write it.

The WIRE framework

Four elements that narrow ambiguity and strengthen intent. Not every prompt needs all four — use what's relevant.

W
What (the task)
What are you asking Claude to build, change, or review? Be specific about the deliverable. "Build a responsive card component" not "make a card."
I
Input (context)
What should Claude reference? Screenshots, existing files, design tokens, component patterns. "Follow the card pattern in components/Card.tsx" or paste a screenshot.

The 8 traps

The most common prompting mistakes designers make — and the fix for each one.

blur_on
Too vague
"Make it look clean" produces random results. Claude can interpret "clean" a hundred ways. Fix: Replace aesthetic adjectives with specific attributes — spacing values, font sizes, color tokens, layout rules.
stacks
Overloading one prompt
"Build the entire page with nav, hero, features, testimonials, and footer" produces a fragmented result. Fix: One component or section per prompt. Build incrementally, just like you'd design in Figma — frame by frame.
block

Expecting the first result to be the final result. No designer ships the first draft. Fix: Prompt, review the output visually, then refine. Each prompt is like a commit in your design process.
hide_source

psychology_alt

Asking Claude to build something you haven't fully thought through. If you can't describe it clearly, Claude can't build it well. Fix: Spend 2 minutes clarifying what you want before typing. Sketch it mentally or on paper first.

01
Exploring — be deliberately open
"Show me three different approaches for a pricing page layout." Overly specific prompts during exploration echo your instructions back instead of surfacing possibilities you hadn't considered.
02
Refining — add constraints incrementally
"I like approach 2. Keep the card layout but switch to horizontal cards on desktop. Use our pricing tier tokens." Narrow down from the exploration one decision at a time.
03
Producing — be highly specific
"Build the final pricing section with three tiers, using our card component, tier tokens from tokens/pricing.css, monthly/annual toggle, and the CTA pattern from the homepage." Precision eliminates ambiguity.
Name visual styles explicitly. "Neobrutalist" produces distinctive output; "trendy" produces nothing. "Editorial with generous whitespace" works; "modern and clean" doesn't. Build a vocabulary of concrete aesthetic references.
Session Management

Managing conversations

How you manage the conversation itself is the third layer of context engineering. A focused, well-managed session produces dramatically better output than a sprawling one — even with identical prompts.

One task, one session

Jumping between unrelated tasks — building a nav, then debugging a form, then writing copy — fills the context window with noise. Each topic dilutes the next. Use /clear to reset between distinct tasks, or open parallel sessions with claude in another terminal tab.

Compact before Claude does

When the context window fills up, Claude auto-compresses earlier conversation — and decides what to keep. Take control by compacting manually:

/compact Keep the component structure, design tokens, and the spacing
decisions we made. Drop the early exploration and the approaches
we rejected.

After compacting, press Ctrl+O to review what Claude kept. This takes 30 seconds and prevents the most common quality problem in long sessions.

When to start fresh vs. compact vs. clear

Start fresh
Switching to a completely different project. Output feels generic despite specific prompts. Claude repeats mistakes you already corrected. You've been in the same session for 2+ hours.
Compact
Staying on the same task but the window is filling. You want to keep key decisions and constraints but drop the exploration. You're at 50-60% context and about to do something complex.
Use /clear
Switching tasks within the same session. /clear wipes conversation but keeps CLAUDE.md loaded. Fresh start without re-authenticating or reloading project context.
lightbulb
Commit as you go
Commit to git after every meaningful change. Each commit is a restore point. If a session goes sideways — or you compact and lose something important — you can always roll back. This habit is what separates anxious Claude users from confident ones.

Fundamentals

Skills

Reusable commands that encode your design knowledge so Claude remembers it forever.

Concept

A prompt that graduated

A Skill is a reusable set of instructions saved as a markdown file. Instead of retyping the same prompt every session, you save it once and invoke it with a slash command — /design-review, /build-component, /check-accessibility.

Think of it like onboarding a new team member. You transfer two things: knowledge (your design principles, token specs, brand guidelines) and abilities (how to review, generate, and audit). Skills provide both — and unlike a human, Claude never forgets what you taught it.

01
Persistent
A prompt disappears after one session. A Skill lives in your project files, loads on demand, and works the same way every time you invoke it.
02
Shareable
Skills live in .claude/skills/ and can be committed to git. Your entire team uses the same design review workflow, component generation rules, and audit checklists.
03
Composable
A component-generation skill can reference your design-tokens skill, which references your color-system spec. Update the tokens and everything downstream updates automatically.
04
Context-efficient
Only the skill description loads at session start. The full instructions load only when you invoke it — so you can have dozens of skills without using up Claude's context window.
Decision

Skills vs CLAUDE.md

Both encode your design knowledge, but they serve different purposes. Getting this distinction right keeps your system lean and effective.

description
CLAUDE.md — always-on rules
Loads every session, automatically. Keep it short (under ~100 lines). Put project conventions, code style rules, tech stack, and things Claude must always follow here. For every line, ask: "Would Claude make a mistake without this?"
build
Skills — on-demand workflows
Loads only when invoked. Use for detailed procedures, multi-step workflows, reference material, and tasks that accept arguments. No length penalty when not in use — go as detailed as needed.
lightbulb
The rule of thumb
If Claude needs to know it for every task → CLAUDE.md. If Claude needs to know it for a specific task → Skill. Moving detailed workflows out of CLAUDE.md and into Skills is the #1 way to improve Claude's consistency across all tasks.
Structure

Anatomy of a Skill

A Skill is a markdown file with a YAML header and instructions. Here's what each part does.

---
name: design-review
description: >
  Review UI components and pages against our design system.
  Trigger when: user asks to review, audit, or check a component.
---

Review the component or page described in $ARGUMENTS.

## Steps
1. Read the target file(s) and our design tokens
2. Check all colors use token variables (no hardcoded hex values)
3. Check spacing follows our 4px grid using --sds-space-* tokens
4. Check typography uses our type scale
5. Check all interactive elements have hover, focus, and active states
6. Check responsive behavior at 375px, 768px, and 1280px

## Output format
Return a markdown checklist:
- ✅ Passing items with brief confirmation
- ❌ Failing items with the specific issue and suggested fix

## Gotchas
- Don't flag third-party library styles (node_modules)
- Our brand blue is --sds-color-blue-750, not --sds-color-blue-500

Key elements

badge
Frontmatter — name and description
The name becomes the /slash-command. The description tells Claude when to auto-load the skill — front-load the key use case and include trigger words.
data_object
$ARGUMENTS — dynamic input
Whatever you type after the command name replaces $ARGUMENTS. So /design-review src/Header.tsx passes the file path into the skill instructions.
format_list_numbered
Steps — the procedure
Numbered steps Claude follows in order. Use imperative verbs ("Read," "Check," "Return"). This is the core of the skill — be specific about what to examine and in what order.
output
Output format — what the result looks like
Explicitly define the format: markdown checklist, JSON, HTML, a code file. Include an example. Without this, Claude guesses — and you spend time reformatting every response.
warning
Gotchas — learned exceptions
Document failure modes as you discover them. When Claude makes a mistake while running the skill, add it here. Over time, this becomes the highest-value section of the file.

Responsive audit

Check a component or page at every breakpoint and flag layout issues. Pair this with your design review skill for a complete quality pass.

---
name: responsive-audit
description: >
  Audit responsive behavior of a component or page across
  breakpoints. Trigger when: user asks to check responsive,
  test breakpoints, or audit mobile/tablet layout.
---

Audit the responsive behavior of $ARGUMENTS.

## Breakpoints to check
- 375px  (mobile — iPhone SE)
- 390px  (mobile — iPhone 14)
- 768px  (tablet — iPad portrait)
- 1024px (tablet — iPad landscape)
- 1280px (desktop — standard)
- 1440px (desktop — large)

## What to check at each breakpoint
1. Does content reflow without horizontal scrolling?
2. Are touch targets at least 44x44px on mobile?
3. Does the navigation adapt (hamburger, collapsed, etc.)?
4. Are font sizes readable without zooming?
5. Do images and cards stack appropriately?
6. Is spacing proportional (not too cramped or too loose)?
7. Are any elements hidden that shouldn't be, or vice versa?

## Output format
Return a table: breakpoint | status | issues found.
Flag critical issues (broken layout, unreachable content)
separately from minor issues (spacing tweaks, alignment).

Content review

Audit all user-facing copy against your voice and tone guidelines. Catches inconsistent button labels, jargon in error messages, and missing microcopy.

---
name: content-review
description: >
  Review UI copy for voice, tone, clarity, and consistency.
  Trigger when: user asks to review copy, check content,
  audit text, or improve microcopy.
---

Review all user-facing text in $ARGUMENTS.

## What to check
1. **Button labels** — action verbs, consistent casing
   (e.g., always "Save changes" not "Submit" or "OK")
2. **Headings** — sentence case, clear hierarchy,
   no marketing-speak in product UI
3. **Error messages** — explain what happened, what to do
   next, no blame ("We couldn't save" not "Invalid input")
4. **Empty states** — helpful, suggest an action, not just
   "No data" or "Nothing here"
5. **Placeholder text** — real examples, not "Enter text here"
6. **Tooltips / help text** — concise, answers one question
7. **Consistency** — same action = same label everywhere

## Tone rules
- Professional but approachable
- Direct, not clever
- Second person ("you" not "the user")
- No jargon unless the audience expects it

## Output format
Return a markdown checklist grouped by element type.
Include the current text, the issue, and a suggested rewrite.

UX Research

How to use Claude Code across the full research pipeline — from raw transcripts to stakeholder-ready insights.

The Opportunity

Why research changes first

80% of researchers now use AI in their workflow. Research involves large volumes of unstructured text — exactly what LLMs excel at processing.

Research is where AI delivers the most immediate, tangible value for design teams. Not because the tools are better for research, but because the raw material of research — transcripts, survey responses, field notes — is text. And text is AI's native format.

01
The volume problem
A single research study generates tens of thousands of words across transcripts, notes, and survey responses. Manually processing this volume forces shortcuts — you skim instead of read, you code the obvious themes and miss the subtle ones. AI processes every word with equal attention.
02
Text is the native format
Unlike visual design — where AI must interpret spatial relationships and aesthetics — research data is already in the format LLMs understand best. Transcripts, open-ended survey responses, and field notes are pure text. No translation layer needed.
03
Pattern recognition at scale
Humans are excellent at recognizing patterns in small datasets but lose fidelity as volume grows. AI maintains consistency across hundreds of responses. It doesn't get tired at transcript seven. It doesn't anchor on the most articulate participant.
04
Researcher judgment remains
AI handles the mechanical work — extraction, sorting, counting, initial grouping. You handle the interpretive work — deciding what matters, why it matters, and what to do about it. The division is clean and the value of your judgment actually increases.
"AI does the mechanical work so I can focus on interpretation." This is the reframe. You're not being replaced — you're being promoted from data processor to insight strategist.
The Framework

The research pipeline

Five steps from raw data to actionable insight. AI's trustworthiness varies at each step — know where to lean on it and where to hold the reins.

mop
1. Clean — Prepare raw data
Strip PII, fix transcription errors, normalize formatting, add participant labels. Trust level: High. This is mechanical transformation with clear right/wrong answers. Let AI handle it fully, then spot-check.
label
2. Tag — Extract and label observations
Pull out key quotes, behaviors, pain points, and feature requests. Tag each with participant ID and context. Trust level: Medium-high. AI is good at extraction but may miss sarcasm, context-dependent meaning, or culturally specific references.
workspaces
3. Cluster — Group related observations
Organize tagged observations into thematic groups. Identify patterns across participants. Trust level: Medium. AI proposes groupings; you validate whether they reflect real patterns or superficial similarity. Ask for multiple clustering axes.
psychology
4. Synthesize — Derive insights
Turn clusters into findings, implications, and recommendations. Trust level: Low-medium. This requires domain expertise, business context, and design judgment. AI drafts; you rewrite, reframe, and validate against what you actually observed.
present_to_all
5. Communicate — Deliver to stakeholders
Frame insights for your audience — executives, engineers, product managers. Trust level: Low. Communication is persuasion, not summary. You know the politics, the priorities, and what will actually get built. AI can structure; you must frame.
lightbulb
Trust varies by step
Cleaning is high trust — let AI run. Synthesis requires heavy human judgment — AI drafts, you decide. Understanding this gradient is the difference between using AI effectively and using it dangerously.
bookmark
The most dangerous AI output
Skipping "clean" and jumping straight to "synthesize" produces plausible-sounding insights from messy data. The output looks polished. The reasoning is confident. And it's built on a foundation of transcription errors, mislabeled participants, and uncleaned noise. Always run the pipeline in order.
Judgment Calls

What stays human

AI can process research data. It cannot do research. These are the judgment calls that remain yours.

Letting AI name themes
→
Validating themes against your domain expertise
Theme validation requires understanding the problem space, the user population, and what has already been tried. AI groups by textual similarity; you group by meaning. A theme called "navigation issues" might actually be three distinct problems.
Using AI summaries in presentations
→
Framing insights for your specific audience
Communication is persuasion, not summary. You know which stakeholder needs the business case, which one needs the user story, and which one just needs the recommendation. AI writes for a generic audience — you write for the people in the room.
Generating personas from prompts
→
Tracing every persona claim to observed behavior
Personas must be grounded in data, not plausibility. AI personas feel real because they are coherent — but coherence is not accuracy. Every claim in a persona should point back to a specific participant quote or observed behavior.
Accepting AI's assessment of relevance
→
Applying your own design judgment to what matters
Relevance is a design judgment. AI cannot know what your team is capable of building, what the business will fund, or which insight will unlock the next product decision. That filter is yours.
error_outline
The biggest risk is false confidence
Clean output from bad data looks trustworthy. AI-generated insights are articulate, well-structured, and presented with confidence — even when the underlying data is thin, biased, or misinterpreted. The more polished the output looks, the more carefully you need to verify the inputs.
Workflow

Transcript analysis

The most common research workflow with Claude — turning hours of interview transcripts into structured, actionable observations.

looks_one
Strip PII
Remove names, email addresses, company names, and any identifying details before feeding transcripts to Claude. Replace with participant codes: P01, P02, etc. This is a legal and ethical requirement, not optional.
looks_two
Add participant labels
Tag each transcript with participant metadata: role, segment, recruitment criteria, session date. This context helps Claude distinguish patterns by user type rather than treating all participants as identical.
looks_3
First-pass extraction
Ask Claude to pull out key quotes, behaviors, pain points, workarounds, and feature requests. Request that each observation includes the participant code, a direct quote, and the context around the quote.
looks_4
Review against transcript
Read Claude's extraction alongside the original transcript. Check for missed observations, misinterpreted context, and quotes pulled out of context. This is where your judgment matters most — AI captures words, you capture meaning.
looks_5
Cross-participant synthesis
After reviewing individual extractions, feed the validated observations to Claude and ask for patterns across participants. Where do three or more participants report the same behavior? Where do segments diverge?
Read the attached interview transcript for participant P03
(enterprise admin, 8 years experience, onboarded 6 months ago).

Extract the following, with direct quotes for each:
1. Pain points — what frustrated them or slowed them down
2. Workarounds — processes they invented because the tool didn't support their need
3. Feature requests — anything they explicitly asked for
4. Positive moments — what they praised or said works well
5. Unexpected behaviors — anything that surprised you as the analyst

For each observation, include:
- Participant: P03
- Category: [pain point / workaround / request / positive / unexpected]
- Quote: "[exact words]"
- Context: [what was happening in the conversation when they said this]
- Severity: [blocker / major / minor / enhancement]
lightbulb
Feed transcripts one at a time, not all at once
Processing transcripts individually forces Claude to give each participant full attention. Feeding all transcripts together causes Claude to anchor on the most articulate or extreme voices and under-represent quieter participants. Process individually, then synthesize across.
Workflow

Affinity mapping

Turn a flat list of observations into structured groupings — and ask for multiple clustering axes to avoid locking into one perspective too early.

Traditional affinity mapping involves sticky notes on a wall and hours of manual sorting. Claude can propose initial groupings in seconds — but the real value is asking for multiple clustering axes. Instead of one organization, get three or four. Each reveals different patterns in the same data.

The key technique: don't just ask Claude to "group these observations." Ask it to group them by task, then by pain point type, then by user segment. The observations that appear in the same cluster across all three axes are your strongest findings. The ones that shift between clusters reveal nuance.

bookmark
Outliers are often the most important findings
The observations that don't fit neatly into any cluster are the ones worth investigating. They might represent an emerging need, an edge case that affects a critical user segment, or a problem so novel that no one else has encountered it yet. Don't let AI bury them in an "Other" category.
Here are 47 validated observations from 8 participant interviews
(see attached). Each has a participant code, category, quote,
and context.

Cluster these observations using THREE different axes:

**Axis 1 — By user task**
Group by the task the participant was trying to accomplish
when they made this observation.

**Axis 2 — By pain point type**
Group by the nature of the friction: usability, performance,
missing feature, workflow mismatch, or conceptual confusion.

**Axis 3 — By user segment**
Group by participant type (enterprise admin vs. team member
vs. new user) and note where segments agree or diverge.

For each axis, list:
- Cluster name and observation count
- Top 3 supporting quotes
- Observations that DON'T fit any cluster (outliers)

Finally: which observations appear in the strongest cluster
across all three axes? These are your highest-confidence findings.
Workflow

Survey synthesis

A two-pass approach that keeps the researcher in control of the codebook while letting AI handle the volume.

looks_one
Pass 1 — Generate codebook from a sample
Feed Claude a random sample of 20-30 open-ended responses. Ask it to propose a codebook: a set of themes with definitions and example quotes for each. This is a draft — Claude proposes, you refine.
looks_two
Researcher adjusts the codebook
Review the proposed themes. Merge overlapping categories. Split overly broad ones. Rename themes to match your research framework. Add themes Claude missed because they require domain knowledge. This is where your expertise shapes the analysis.
looks_3
Pass 2 — Claude applies codebook to all responses
Feed Claude your refined codebook and all remaining responses. Ask it to tag each response with one or more themes, a confidence level, and a flag for responses that don't fit any theme. Request counts for each theme.
looks_4
Audit a random sample
Randomly select 15-20 responses and verify Claude's coding against your own judgment. Calculate agreement rate. If below 80%, refine the codebook definitions and re-run. This is your quality control step.

Starting Point

Design exploration

The value of AI in early design isn't getting to a final answer faster — it's expanding the possibility space before you commit to a direction.

Most designers start a project with one or two layout ideas in mind and refine from there. Claude changes the math. Instead of exploring two directions deeply, you can explore eight directions quickly — then apply your judgment to the one worth refining. The point is not to accept Claude's output. It's to see options you wouldn't have considered.

01
Diverge before you converge
Ask Claude for three to five layout variations for the same content. Don't over-specify — give it the content, the constraints, and the design system, then let it generate options. Your job is to spot the promising direction, not to prescribe it upfront.
02
Use constraints, not solutions
Instead of "make a card grid with 3 columns," say "this content has 6-12 items, needs to scan quickly, and must work on mobile." Constraints give Claude room to propose layouts you wouldn't have considered. Solutions just make it a typist.
03
Explore visual tone
Ask Claude to generate the same component with different visual treatments: dense vs. spacious, minimal vs. detailed, flat vs. layered. Seeing the same content in different presentations helps you articulate what you actually want — which is often hard to do in the abstract.
04
Throw away most of it
Exploration output is disposable. The goal is to inform your thinking, not to ship Claude's first attempt. If you generated five directions and kept elements from two of them, that's a successful exploration — even though you "wasted" three generations.
lightbulb
Exploration prompts should be loose, not tight
The more specific your prompt, the narrower the output. During exploration, give Claude the problem and constraints but not the solution. Save your detailed specs for the component creation phase — exploration is about breadth, not precision.

Responsive design

Responsive layout is largely mechanical — and that makes it one of the highest-leverage workflows for Claude. Let AI handle the breakpoints while you focus on the experience at each size.

Making a layout work across viewports involves predictable decisions: stack columns, resize type, adjust spacing, hide secondary content. Claude handles this mechanical translation well. The design judgment — what gets priority on mobile, how navigation adapts, which content collapses — stays with you.

01
Design the desktop, describe the mobile intent
Build or prompt for the desktop layout first, then tell Claude how you want it to adapt: "On mobile, stack the two-column layout, prioritize the data table over the chart, and collapse the filter panel into a bottom sheet." Intent-based responsive direction beats pixel-specific breakpoint rules.
02
Test with the preview tool
Use Claude's preview tool to resize the viewport and verify layouts at each breakpoint. Catching responsive issues in real-time — during the generation session — is dramatically faster than discovering them after handoff. Check mobile, tablet, and desktop widths.
03
Trust Claude for the math, not the priority
Claude is reliable at converting multi-column layouts to single-column, adjusting grid gaps, and scaling typography. It is less reliable at deciding what to show, hide, or rearrange for smaller screens. Those are content strategy decisions — provide them explicitly.
04
Use your design tokens for breakpoints
If your design system defines breakpoint tokens, tell Claude to use them. Consistency with your existing responsive system matters more than Claude's opinion about where breakpoints should fall. Your tokens encode decisions your team already made.
error_outline
Always check the in-between sizes
Claude tends to optimize for the exact breakpoints you specify — 375px, 768px, 1280px. Real users hit every width in between. Resize the preview slowly across the full range and look for awkward intermediate states where the layout breaks. These in-between widths are where responsive bugs hide.

Accessibility audits

Accessibility checking is the most reliable, highest-trust AI workflow for designers. The rules are codified, the criteria are measurable, and the output is verifiable.

Unlike visual design or content strategy — where AI output requires subjective judgment — accessibility has clear pass/fail criteria. WCAG guidelines are specific. Color contrast ratios are mathematical. ARIA roles are either correct or incorrect. This makes accessibility auditing the workflow where you can trust Claude the most and verify the fastest.

Claude can predict what a screen reader will announce for each element and flag confusing or missing announcements. "Button: X" is not helpful — "Button: Close notification" is. This review catches the difference between technically accessible and actually usable.

Iteration patterns

How to iterate effectively with Claude: when to refine in-session, when to start fresh, and how to give corrections that actually improve the output.

Most designers either over-iterate (spending 20 rounds of corrections on diminishing returns) or under-iterate (accepting the first output because it looks "close enough"). Effective iteration with Claude follows predictable patterns — learn them and your output quality jumps significantly.

Vague corrections: "make it better"
→
Specific corrections: "increase heading size to --sds-font-heading-lg and reduce card padding to --sds-space-300"
Claude cannot read your mind. "Make it better" might produce a change you didn't want, triggering another round of correction. Specific feedback — with token names, pixel values, or behavioral descriptions — produces the change you intended on the first try.
Iterating forever in one session
→
Starting fresh after 3-4 rounds if quality is declining
After many iterations, Claude's context becomes cluttered with old instructions and corrections. If you notice output quality degrading or Claude reverting previous fixes, start a new session. Paste in the current code and your remaining corrections — a fresh context often produces better results than continued iteration.
Correcting one thing at a time
→
Batching related corrections into a single prompt
Each round of iteration has overhead. If you need to fix spacing, color, and typography in the same component, describe all three changes in one prompt rather than three separate messages. Batching produces more coherent updates and keeps your session context cleaner.
Trying to fix a structural problem with surface edits
→
Recognizing when you need a different approach entirely
If the layout concept is wrong, no amount of spacing and color correction will fix it. Recognize the difference between refinement (the approach is right, the details need adjustment) and pivot (the approach is wrong, start over). Pivoting early saves more time than perfecting the wrong direction.
lightbulb
The 3-round rule
If you haven't converged on something good within three rounds of iteration, the problem is usually the prompt — not the corrections. Go back to your initial spec, identify what was missing or ambiguous, and rewrite it. A better starting prompt beats five rounds of incremental correction.

Working with AI

What actually changes when AI enters your design practice — your role, your workflow, your team, and how to measure progress.

The Real Barrier

The identity question

The biggest barrier to AI adoption isn't skill — it's identity. Senior designers resist AI not because they can't learn it, but because it threatens something deeper: their sense of what makes them valuable.

01
The skill-identity gap
Most designers can learn the tool in a week. But adopting it means redefining what "being a designer" means to them. That's not a training problem — it's a change management problem.
02
The expertise paradox
The most experienced designers often resist the hardest. They've built their careers on craft skills that AI now accelerates. The more senior you are, the more identity is at stake.
03
The craft question
If Claude can generate a component in seconds that took you hours, does the craft still matter? Yes — but the craft shifts from production to direction. You're still the designer. The medium changed.
04
The reframe
AI doesn't replace your judgment — it gives you more surface area to apply it. Instead of spending 4 hours on layout, you spend 30 minutes directing Claude and 3.5 hours on higher-order decisions.
Your value was never in execution speed — it was in knowing what to execute. The designers who thrive with AI are the ones who already had strong opinions. The tool just lets them move faster.
bookmark
This is a people problem, not a tools problem
Teams with strong change management see 6x higher AI adoption success rates. The difference isn't better prompts or fancier tools — it's psychological safety, visible leadership use, and honest conversations about what's changing. Address the identity question first. The skills follow.

Will AI replace designers? No. But designers who use AI will replace designers who don't. The question isn't whether to adopt — it's whether you shape how it enters your practice, or let it happen to you.

Claude gets you to 60% — fast. Structure, layout, component assembly, responsive behavior, first-draft copy. But that last 40% is where the design actually lives. Understanding this split is the most important expectation to set.

The 60% — what AI handles
Scaffolding and structure
Component structure, responsive layouts, boilerplate markup, variant generation, accessibility attributes, first-draft copy, documentation, consistent spacing and token application. This is the mechanical layer.
The 40% — what stays yours
Judgment and refinement
Visual refinement, brand voice tuning, edge case handling, emotional design, micro-interactions, the gap between "technically correct" and "actually good." This is the judgment layer.
AI does the whole thing
→
AI does the scaffolding, you do the refinement
Claude builds the structure. You apply the taste. Neither part works without the other — but the judgment layer is where design quality actually lives.
60% done means 60% time saved

Working Effectively

Session awareness

Claude sessions aren't infinite. Understanding how they work — and how they degrade — is one of the most practical things you can learn. It directly affects the quality of what you get back.

Context windows have limits

Every Claude session has a context window — the total amount of information Claude can hold in memory at once. Think of it like a whiteboard: as the conversation grows, older content gets erased to make room for new content. Claude doesn't choose what to forget — earlier context simply falls off the edge.

This means a session that starts sharp can gradually lose track of decisions you made earlier. You might notice Claude re-asking questions you already answered, contradicting earlier choices, or forgetting constraints you established at the start.

error_outline
Signs your session is degrading
Claude repeats suggestions you already rejected. It forgets design tokens or naming conventions you established. It starts contradicting decisions made earlier in the conversation. Output quality drops noticeably — responses feel more generic or less precise. These are all signals to start fresh.

Large tasks invite hallucinations

When you ask Claude to tackle something too large in a single prompt — "build me a complete design system" or "create a full 10-page prototype" — the scope exceeds what it can hold in working memory. The result: Claude starts making things up. It will invent token names that don't exist, reference components it never built, or generate plausible-looking code that doesn't actually work.

This isn't Claude being unreliable — it's a predictable consequence of asking for too much at once. The fix is simple: break large tasks into smaller, sequential steps. Build one screen, verify it, then build the next. Create the token layer, confirm it, then apply it to components.

Risky
One massive prompt
"Build a complete payment flow with 5 screens, all interactive, with design tokens, transitions, loading states, error handling, and responsive behavior."
Reliable
Sequential steps with checkpoints
"Build the balance screen first. Use these tokens. I'll review, then we'll move to the amount screen." Each step is verified before moving on.

Practical habits for session management

1
Use /compact deliberately
When a session gets long, use /compact to summarize and free up context space. Tell Claude what to keep: "Keep the token names, component structure, and spacing decisions. Drop the early exploration." This extends a session's useful life.
2
Start fresh sessions for new tasks
Don't reuse a prototyping session for a documentation task. Each new task gets a clean session with a focused CLAUDE.md context. Starting fresh is cheap — debugging a confused session is expensive.
3
Verify as you go, not at the end
Review Claude's output after each step, not after ten steps. Catching a hallucinated token name on step 2 is a quick fix. Catching it on step 10 means unwinding everything built on top of it.
4
Write decisions down outside the session
Use CLAUDE.md and DESIGN.md to capture key decisions. These files persist across sessions — so when you start fresh, Claude reads them and picks up where you left off without you re-explaining everything.
lightbulb
The 3-4 round rule
If you've gone 3-4 rounds of iteration on the same element and quality is declining rather than improving, stop iterating. Start a new session, paste in the current state of the file, and give Claude fresh context. You'll almost always get better results than pushing through a degraded session.

Why this matters

When to use this workflow

Competitive research is one of the highest-leverage activities a designer can do — and one of the least done. Not because it isn't valuable, but because it takes too long. Claude changes the economics of research. What took days now takes minutes.

1
Entering a new product area
You need to understand existing patterns, user expectations, and where the bar is set before you start sketching.
2
Evaluating competitive position
You need an honest assessment of where you stand relative to competitors. What are they doing that you aren't? Where are you ahead?
3
Building a design strategy
You need evidence that the current experience is falling behind and a clear picture of the opportunity.
4
Justifying a redesign
Competitive evidence is the fastest way to make the case without relying on opinion.
Step 1

Define the research scope

The quality of your competitive research is directly proportional to the quality of your brief. Before you prompt Claude, get clear on four things: product category, target user segments, core job-to-be-done, and constraints like geography and company size.

I'm a product designer working on a cloud data management platform. Our primary users are data engineers and analytics engineers at enterprise companies (1000+ employees).

Help me map the key players, their design approaches, and where users are currently underserved. Start by identifying the categories of competitors I should be looking at.

2

Platforms that overlap with your space but serve a broader purpose. They're building your features into their core — potentially more dangerous than direct competitors.
3

Startups with novel approaches. They may lack market share but signal where user expectations are heading.

Prototyping & Validation

Build interactive prototypes that answer design questions static mockups never could.

Purpose

When to use this workflow

Use this workflow when you need to validate an interaction model, test a flow hypothesis, demonstrate behavior to stakeholders, or answer a question that static mockups can't answer.

Code prototypes are fundamentally different from Figma prototypes. They demonstrate real behavior — filtering that actually filters, forms that track state, tables that sort and paginate.

Foundation

Define what you're validating

Every prototype should answer a specific question. Not "build me the settings page" but "can a user configure pipeline permissions in under 3 clicks?"

Vague brief
"Build a prototype of the data health dashboard."
No clear question. No definition of done. Feedback will scatter across layout, color, copy, and interaction.
Hypothesis-driven
"Does progressive disclosure reduce cognitive load on the dashboard?"
Clear question. Clear interaction to test. Feedback focuses on whether progressive disclosure works.

Example hypotheses worth prototyping

?
Users can configure row-level security policies in under 4 steps
Prototype needs: a policy builder with scope selection, rule definition, preview, and apply steps. Track step count and whether users backtrack or abandon mid-flow.
?
Progressive disclosure reduces time-to-insight on the data health dashboard
Prototype needs: two versions of the dashboard — one flat, one using expandable sections and drill-down. Measure how quickly users identify the three most critical issues.
?
A guided remediation flow reduces mean-time-to-resolve for data quality issues
Prototype needs: an alert detail view that surfaces root cause, suggests a fix, and lets users apply it in-context. Compare against the current workflow of switching to a separate config screen.
?
Inline column-level lineage reduces support tickets for impact analysis
Prototype needs: a column detail panel with upstream/downstream lineage rendered as a mini-graph. Test whether users can answer "what breaks if I change this column?" without leaving the page.
Fidelity

Scope the fidelity

Not every prototype needs to look production-ready. Most should land at mid-fidelity — styled enough to feel real, not so polished that feedback fixates on visual details.

Lo
Low fidelity
Wireframe-quality. Validating structure, flow sequence, basic layout. Internal design team reviews. Build time: minutes.
Mid
Mid fidelity
Styled with tokens and realistic data. Your default for stakeholder walkthroughs and most validation. Build time: hours.
Hi
High fidelity
Near-production. Only for usability testing with external users or executive presentations. Be careful — invites nitpick feedback.
Construction

Prompt Claude with the validation question, constraints, real data structures, and every state. Build screen by screen — verify each before moving to the next.

- Step 3: Assign user groups (searchable multi-select)

Principles

Prototyping principles

These principles keep prototyping fast, focused, and valuable.

1
Start with the interaction, not the layout
Get the behavior working in a plain container first, then dress it up if needed.
2
Use real data structures
Real field names, real entity relationships, real cardinality. The data shape is the design constraint.
3
Prototype the states, not just the happy path
Empty, error, loading, overflow. These states are where interaction models break — and where the real design work happens.
4
Make it disposable
If you're afraid to throw it away, you've over-invested. The value lives in the insight, not the artifact.
5
One prototype per question
Compound prototypes produce ambiguous results. Keep them small, focused, and single-purpose.

Dimension Coverage Gaps Found
Visual States Hover, active, disabled, selected row styles documented Loading skeleton missing; focus ring not visible on row selection; no empty illustration specified

What does an inline-editable cell look like when the row is selected, the user doesn't have edit permission, and the content overflows? You don't need to design every combination, but you need rules for how states compose.

Example: State composition matrix for inline-editable table cell

BASE STATES
───────────────────────────────────────────────────
selected      Row highlight active (background: --sds-surface-selected)
editing       Cell shows text input, value is mutable
error         Validation failed, destructive border + inline message
disabled      Cell is read-only, content at 40% opacity, no pointer events
loading       Skeleton pulse replacing cell content, no interaction

COMPOSED STATES
───────────────────────────────────────────────────
selected + editing
  → Show text input inside the cell WITH the selected row highlight
    retained on the parent row. Input border uses --sds-border-focus.
    Clicking outside the cell confirms the edit AND deselects the row.

selected + disabled
  → Show selected row highlight but render cell content at 40% opacity.
    Row remains in selection count. No edit affordance on hover.
    Checkbox (if present) is checked but visually dimmed.

editing + error
  → Keep the text input open. Replace the focus border with
    --sds-border-destructive. Show inline error message below the
    input within the cell. Pressing Enter re-validates; pressing
    Escape reverts to the last valid value and exits edit mode.

selected + editing + error
  → Combine all three: selected row highlight on the row,
    destructive border on the input, inline error visible.
    Row remains selected regardless of edit outcome.

ILLEGAL COMBINATIONS
───────────────────────────────────────────────────
disabled + editing    → Cannot occur. Disabled cells reject edit entry.
loading + editing     → Cannot occur. Loading cells are non-interactive.
loading + error       → Cannot occur. Validation only runs post-load.
Step 3

Write interaction specifications

Developers don't struggle with pixel measurements — design tokens handle that. They need behavior descriptions: what happens on click, hover, escape, outside-click, tab, and screen reader focus.

bookmark
Write specs for the hard interactions first
Simple button clicks don't need elaborate specs. Inline editing, drag-and-drop, bulk operations with mixed permissions — these need detailed behavioral documentation. Spend time where ambiguity is highest.

Example: Interaction spec for inline-edit cell (DataTable)

COMPONENT: DataTable — Inline-Edit Cell
SCOPE:     Editable text cells within a data table row
────────────────────────────────────────────────────────

1. CLICK BEHAVIOR
   • Single click on an editable cell: selects the parent row.
     Cell shows a subtle pencil icon (trailing, 16px) on hover
     to indicate editability.
   • Double-click on the cell: enters edit mode. Cell content is
     replaced by a text input pre-filled with the current value,
     text fully selected. Pencil icon is hidden during editing.

2. KEYBOARD — EDIT MODE
   • Enter: commits the current input value and exits edit mode.
     Focus returns to the cell in its display state.
   • Escape: discards changes, reverts to the previous value,
     and exits edit mode. Focus returns to the cell.
   • Tab: commits the current value AND moves focus to the next
     editable cell in the row (left-to-right). If no more editable
     cells exist in the row, focus moves to the first editable
     cell of the next row.
   • Shift+Tab: same as Tab but moves to the previous editable cell.

3. KEYBOARD — NAVIGATION (non-edit mode)
   • Arrow keys navigate between cells in the grid pattern.
   • F2 or Enter on a focused editable cell: enters edit mode
     (equivalent to double-click).

4. FOCUS BEHAVIOR
   • On edit entry: input receives a 2px focus ring using
     --sds-border-focus. Parent row retains its selected state.
   • On edit exit: focus ring moves to the cell container.
     The cell is included in the tab order as a single stop.

5. VALIDATION & ERROR HANDLING
   • Validation runs on commit (Enter, Tab) and on blur.
   • If validation fails: cell stays in edit mode, input border
     changes to --sds-border-destructive, and an inline error
     message appears below the input (max 1 line, truncated
     with tooltip if necessary).
   • The user must fix or Escape. Tabbing away from an invalid
     cell is blocked — focus remains until resolved or reverted.

6. SCREEN READER ANNOUNCEMENTS
   • On edit entry: announce "Editing [column name], current
     value [value]."
   • On successful commit: announce "[Column name] updated
     to [new value]."
   • On validation error: announce "Error: [error message].
     Press Escape to revert."
   • On Escape revert: announce "[Column name] reverted to
     [original value]."
Step 4

Document edge cases

In data management SaaS, edge cases aren't edge cases — they're the product. Zero rows, 10,000 rows, 500-character column values, partial permissions, network drops mid-save.

Handoff that sticks

A spec nobody reads is worse than no spec — it gives the illusion that engineering has what they need.

1
Write specs in the code repo
Not Figma comments, not Confluence, not Slack. Put DESIGN.md next to the component code. Engineers live in the repo — meet them there.
2
Run the spec past an engineer first
They'll catch implementation gaps you missed. A 15-minute review saves days of back-and-forth.
3
Update when requirements change
A stale spec is worse than no spec. Treat DESIGN.md like code — if it's out of date, it's a bug.
4
Handoff is a conversation, not a document drop
Walk through the spec with the implementing engineer. The DESIGN.md is the record of that conversation — not a replacement for it.
