Pass 1 — Full WCAG 2.1 AA Audit

Initial audit identified 10 failures across perceivable, operable, understandable, and robust criteria.

Pass 1 · 10 findings
Risk meter bars convey information by color alone 1.4.1Level A
Critical
The risk meter on the Approval Queue uses green, yellow, and red colored bars with no text alternative. Users who cannot perceive color cannot determine the risk level.
Fix applied: Added a text label next to the risk meter ("Low", "Medium", "High") and an aria-label on the container. Also added pattern fills (solid, diagonal, cross-hatch) as a secondary visual encoding.
WCAG 1.4.1 Use of ColorFixed
Status tags rely solely on color 1.4.1Level A
Critical
Status tags (Pending, In Review, Approved, Denied, Provisioned) use color coding without any other differentiator. Screen reader users only hear the text, but low-vision users who can see some color but not distinguish hues are affected.
Fix applied: Added a leading icon to each status tag: clock for Pending, eye for In Review, check_circle for Approved, cancel for Denied, verified for Provisioned. Icons serve as a shape-based differentiator.
WCAG 1.4.1 Use of ColorFixed
Sensitivity badge contrast fails for yellow-on-white 1.4.3Level AA
Major
The "Medium" sensitivity badge uses yellow-500 (#8A7515) on yellow-025 (#FCF9D9), yielding a contrast ratio of 3.8:1, below the 4.5:1 AA threshold for small text.
Fix applied: Darkened the text color to yellow-600 (#6E5B09) on the same background, achieving 5.2:1 contrast ratio. Applied the same fix to all warning-themed tags and chips.
WCAG 1.4.3 Contrast (Minimum)Fixed
Tables lack proper header associations 1.3.1Level A
Major
Data tables on My Requests, Approval Queue, and Audit Log use th elements but lack scope attributes. Complex tables may not convey cell-to-header relationships to screen readers.
Fix applied: Added scope="col" to all column headers. Added caption elements with descriptive text (visually hidden with sr-only class). Added aria-sort to sortable columns.
WCAG 1.3.1 Info and RelationshipsFixed
Sidenav links lack current page indication 2.4.8Level AA
Major
The active nav item uses visual styling (background color) but does not convey the current page to screen readers.
Fix applied: Added aria-current="page" to the active sidenav item. Added role="navigation" with a descriptive aria-label to the nav container.
WCAG 2.4.8 LocationFixed
Form inputs lack programmatic labels 1.3.1Level A
Critical
The new request form uses visual labels (.form-label) but the label elements are not linked to their inputs via for/id attributes.
Fix applied: Added unique id attributes to all form controls and linked them to their label elements using for attributes. Also added aria-describedby for hint text.
WCAG 1.3.1 Info and RelationshipsFixed
Icon buttons lack accessible names 4.1.2Level A
Major
Header icon buttons for notifications and help use Material Symbols icons with no aria-label. Screen readers announce them as unlabeled buttons.
Fix applied: Added aria-label to all icon-only buttons ("Notifications", "Help", "User menu"). Added role="img" and aria-hidden="true" to the Material Symbols spans.
WCAG 4.1.2 Name, Role, ValueFixed
Focus indicators missing on custom controls 2.4.7Level AA
Major
Custom checkboxes, filter chips, and screen-nav buttons suppress default focus outlines without providing a custom visible focus indicator.
Fix applied: Added a consistent focus-visible style across all interactive elements: 2px solid outline using --sds-color-blue-750 with a 2px offset. The outline only appears on keyboard navigation (focus-visible), not on mouse click.
WCAG 2.4.7 Focus VisibleFixed
Timeline component lacks semantic structure 1.3.1Level A
Minor
The status timeline is built with divs. Screen readers cannot convey the ordered, sequential nature of the timeline entries.
Fix applied: Wrapped timeline in an ordered list (ol) with each event as a list item (li). Added aria-label="Request status timeline" to the container. Screen readers now announce "list, 3 items" and each step in order.
WCAG 1.3.1 Info and RelationshipsFixed
Page title does not update on screen change 2.4.2Level A
Minor
When navigating between screens in the prototype, the document title stays static. Screen reader users cannot identify which screen they are viewing.
Fix applied: Updated the showScreen() function to dynamically set document.title to include the current screen name (e.g., "My Requests - Access Request Portal"). Added aria-live="polite" announcement for screen transitions.

Pass 2 — Re-Audit After Fixes

Re-audit of fixed designs found 3 new issues introduced by the fixes.

Pass 2 · 3 findings
Status tag icons are not hidden from screen readers 4.1.2Level A
Minor
The new status tag icons (check_circle, cancel, etc.) are announced by screen readers alongside the text, causing redundant announcements like "check circle Approved".
Fix applied: Added aria-hidden="true" to the icon spans within status tags. The text label alone conveys the status to assistive technology.
WCAG 4.1.2 Name, Role, ValueFixed
Risk meter pattern fills reduce readability at small sizes 1.4.11Level AA
Minor
The cross-hatch pattern for the "High" risk bar becomes visually indistinct at the 6px bar height, making it harder to differentiate from the solid pattern.
Fix applied: Increased risk bar height to 8px and simplified patterns: solid fill for Low, diagonal stripe for Medium, dense dot pattern for High. Also increased the bar width from 24px to 28px.
WCAG 1.4.11 Non-text ContrastFixed
aria-live announcement fires too frequently 4.1.3Level AA
Minor
The screen transition announcement fires even when the same screen is re-selected, creating unnecessary announcements for screen reader users.
Fix applied: Added a check to only announce when the screen actually changes. Debounced rapid screen switches to prevent announcement flooding.
