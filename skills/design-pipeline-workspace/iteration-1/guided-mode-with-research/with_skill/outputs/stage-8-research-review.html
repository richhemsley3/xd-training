Research Review

Evaluating designs against original user interview findings · Are we solving the right problems?

87%
Overall Research Alignment Score
The designs strongly address the core pain points identified in user interviews. 5 of 6 pain points are fully addressed, 1 is partially addressed. All 3 personas' primary goals are served. Two opportunities for deeper alignment are identified below.

Pain Point Coverage

Evaluating each research-identified pain point against the current designs.

No visibility into request status
Fully Addressed
The design provides real-time status tracking through multiple touchpoints: status tags on the request list, a detailed status timeline on request detail, and a stat card summary showing active/approved/denied counts. Requestors can see exactly where their request stands at all times.
Design evidence: My Requests list view (status column), Request Detail screen (status timeline component), stat cards showing request counts by status.
Duplicate requests
Fully Addressed
The Duplicate Warning screen directly intercepts duplicate submissions by detecting matching pending requests and surfacing them to the user. This addresses the ~30% duplicate rate identified in interviews and breaks the negative feedback loop of resubmission.
Design evidence: Duplicate Warning screen with callout banner, existing request card, and "View Existing Request" / "Submit Anyway" options.
Approvals getting lost
Fully Addressed
Automated routing eliminates the manual email forwarding that caused requests to be lost. Every request is tracked in the system from submission to resolution, with an immutable audit log. The centralized approval queue ensures approvers see all pending requests in one place.
Design evidence: Auto-routing shown in status timeline, Approval Queue with pending count, Audit Log with complete event history.
3-5 manual handoffs per request
Fully Addressed
The proposed flow reduces handoffs from 3-5 to effectively zero manual handoffs. The system handles routing (auto-route to owner), notification (in-app + configurable), and provisioning (auto-provision after approval). The only human step is the approval decision itself.
Design evidence: New Request form submits directly to system, auto-routing shown in timeline, auto-provisioning mentioned in flow design.
No audit trail
Fully Addressed
The Audit Log screen provides a complete, filterable, exportable record of every access request action. Security admins can search by event type, date range, actor, or dataset. The CSV export capability directly supports compliance audit preparation.
Design evidence: Audit Log screen with event table, filter chips (Approvals, Denials, Provisioning, Revocations), Export CSV button, search input.
Inconsistent access policies
Partially Addressed
The sidenav includes a Policies section and the IA proposes policy management, routing rules, and auto-approval policies. However, the current prototype does not include detailed screens for policy creation, editing, or enforcement visualization. Security admins can see the audit log but cannot yet define or manage the policies that govern access decisions.
Design evidence: Policies nav item exists but no screen is designed. IA proposes Policy Management, Routing Rules, and Auto-Approval Policies screens. Review Request screen shows sensitivity context but does not reference specific policies.

Persona Goal Coverage

Does each persona's primary workflow deliver on their stated goals?

Data Engineer (Requestor)
Goals Met
  • Get access quickly: Self-service form with auto-routing targets under 1 hour for standard requests vs. 5-10 days currently.
  • Understand what data is available: Data Catalog section in sidenav with inline request capability from catalog detail.
  • Avoid administrative overhead: Structured form with searchable dataset picker reduces time spent composing emails and tracking down owners.
Data Owner (Approver)
Goals Met
  • Make informed decisions quickly: Review Request screen shows requestor profile, dataset details, sensitivity, PII field count, and risk level all in one view.
  • Not become a bottleneck: Batch approval for low-risk requests, risk-based triage, and one-click approve/deny reduce review time.
  • Track and revoke access: Approval Queue shows all decisions; future Access Review screen supports periodic certification.
Security Admin (Auditor)
Partially Met
  • Complete audit trail: Fully addressed with the Audit Log screen, filterable and exportable.
  • On-demand access reviews: IA proposes Access Reviews but no screen is designed yet.
  • Enforce consistent policies: Policies nav item exists but screens are not designed. This is the main gap for this persona.

Recommendations for Next Iteration

Based on the gaps identified above, these improvements would strengthen research alignment.

1. Design the Policy Management screens
The "inconsistent access policies" pain point is only partially addressed. Security admins need screens to create, edit, and visualize access policies. This should include: policy templates for common scenarios, a rule builder for routing logic, and a policy conflict detector. This directly addresses the Security Admin persona's primary gap.
High Priority
2. Design the Access Review workflow
The "audit fire drill" pattern from the journey map is addressed by the audit log for historical review, but the periodic certification flow (Flow 8 in the IA) has no designed screens. Security admins need to be able to initiate reviews, track owner responses, and execute bulk revocations. This is a key differentiator from email-based processes.
High Priority
3. Add notification channel integration
The research showed that email is the wrong medium for approvals, yet the current design only shows in-app notifications. Adding Slack integration for approval notifications (with one-click approve from Slack) would meet users where they already work and further reduce the time-to-approval. This aligns with the "embed in existing workflows" principle from the broader product strategy.
Medium Priority
4. Add time-based access expiration
The new request form includes a Duration field, but the design does not show what happens when access expires. Adding expiration reminders, renewal workflows, and automatic revocation would prevent the accumulation of stale access grants that the research identified as a compliance risk.
Medium Priority
5. Deeper Data Catalog integration
The research mentioned Data Engineers need to "understand what data is available and who owns it." The current Data Catalog is a nav item but has no designed screens. A catalog with search, browse by domain, sensitivity labeling, and inline "Request Access" would complete the end-to-end experience from discovery to access.
Low Priority (v2)
