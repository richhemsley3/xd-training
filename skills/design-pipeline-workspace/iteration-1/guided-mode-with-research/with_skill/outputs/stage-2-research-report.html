Executive Summary

Core findings from user interviews with data engineers, data owners, and security admins.

Data engineers spend 40% of their working time managing access request workflows. The current approval process requires 3 to 5 manual handoffs across teams, creating bottlenecks, delays, and frustration. Participants described the experience with strong negative language, indicating deep dissatisfaction with existing tools and processes.

Three pain points dominated every interview: no visibility into where a request stands in the approval chain, duplicate requests created when the original appears lost, and approvals that genuinely get lost in email threads. The primary opportunity is a self-service access request portal with automated routing that eliminates manual handoffs and provides real-time status visibility.

Primary Opportunity: Build a self-service access request portal with automated routing that reduces the 3–5 manual handoffs to zero, gives all stakeholders real-time visibility, and prevents duplicate requests through status transparency.

Key Insights

The most significant patterns from user interviews, ranked by frequency and severity.

Critical

Approval Workflow is the Bottleneck

The current process requires requests to pass through 3–5 teams manually. Each handoff introduces delay, ambiguity about ownership, and risk of the request being dropped entirely.

Critical

No Status Visibility

Requestors have no way to check where their access request stands. They resort to pinging people on Slack, sending follow-up emails, or re-submitting entirely — creating more noise in an already overloaded system.

High

Duplicate Requests Compound the Problem

Because status is opaque, requestors submit duplicate requests, which doubles the workload for approvers and further clogs the pipeline. This creates a negative feedback loop.

High

Email is the Wrong Medium

Approvals live in email threads that get buried, forwarded to wrong people, or simply lost. There is no structured data, no audit trail, and no way to escalate systematically.

Moderate

Audit Trail Gaps

Security admins cannot reconstruct who approved what and when. Compliance audits require manual archaeology through email and Slack, which is both time-consuming and unreliable.

Moderate

Cross-Team Coordination Friction

Different teams use different tools and processes. There is no single source of truth for access policies, which leads to inconsistent enforcement and shadow access grants.

Personas

Three key personas identified from user interviews.

DE

Data Engineer (Requestor)

Primary User

Role

Builds and maintains data pipelines. Needs access to datasets across the organization to build and debug ETL jobs, create reports, and support analytics teams.

Goals

  • Get access to the right data quickly so they can do their primary job
  • Understand what data is available and who owns it
  • Avoid spending time on administrative overhead

Frustrations

  • Spends 40% of time on access request workflows instead of engineering
  • Has no idea where requests stand after submission
  • Often re-submits requests because the original seems lost
DO

Data Owner (Approver)

Key Stakeholder

Role

Responsible for specific datasets or domains. Must approve or deny access requests based on data sensitivity, requestor role, and business justification.

Goals

  • Make informed approval decisions quickly
  • Maintain appropriate access controls without becoming a bottleneck
  • Have confidence that approved access is tracked and revocable

Frustrations

  • Receives requests via email with insufficient context
  • Cannot distinguish new requests from duplicates
  • No easy way to see all pending requests in one place
SA

Security Admin (Auditor)

Governance

Role

Ensures that all data access complies with internal policies, regulatory requirements, and security standards. Conducts periodic access reviews and audit responses.

Goals

  • Complete audit trail of every access grant and revocation
  • Ability to run access reviews on-demand
  • Enforce consistent access policies across the organization

Frustrations

  • No centralized record of who approved what
  • Audit preparation requires manual investigation across email and Slack
  • Cannot verify that access grants match current policy

Patterns

Behavioral patterns observed across interviews.

The Status Check Loop

Requestors submit, wait 1–2 days, then begin actively pinging approvers on Slack. If no response within another day, they re-submit. This creates a cycle of noise that buries legitimate requests.

The Shadow Approval

When formal channels fail, requestors go directly to someone with admin access and ask for a quick grant. This bypasses all governance controls and creates untracked access.

The Batch Approver

Data owners overwhelmed by request volume batch-approve everything once a week rather than reviewing individually. This leads to over-permissioning and compliance risk.

The Audit Fire Drill

Security admins only discover access problems during quarterly audits. They spend days reconstructing approval chains from email, resulting in incomplete records and policy gaps.

Opportunity

The primary design opportunity identified from this research.

Self-Service Access Request Portal with Automated Routing

Replace the email-based, multi-handoff approval process with a centralized portal where requestors can submit, track, and manage access requests, while approvers receive structured requests with full context and one-click approval capabilities.

Recommendations

Design recommendations based on research findings.

Must Have

Real-Time Status Tracking

Every request must have a visible status (submitted, in review, approved, provisioned) accessible to all stakeholders at all times. This is the single most impactful feature to address user frustration.

Must Have

Automated Routing Engine

Route requests to the correct approver(s) based on data classification, domain ownership, and sensitivity level. Eliminate manual forwarding entirely.

Must Have

Duplicate Detection

Detect when a requestor submits a request that matches an existing pending or recently approved request. Surface the existing request instead of creating a new one.

Should Have

Approver Dashboard

Single view showing all pending requests with context (requestor, dataset, sensitivity, justification). Enable batch actions and one-click approvals for low-risk requests.

Should Have

Complete Audit Trail

Immutable log of every action: submission, routing, approval, denial, escalation, provisioning, and revocation. Exportable for compliance audits.

Nice to Have

Policy-Based Auto-Approval

For low-sensitivity datasets with clear policies, enable automatic approval based on the requestor's role and the data classification. Reserve human review for high-sensitivity access.
